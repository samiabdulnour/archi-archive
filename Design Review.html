02 — DIAGNOSIS

Five issues with
the current taxonomy UI.

Listed in order of how much they hurt the "ten-second sidewalk" promise.

Issue 01 of 05

Two questions promised, eleven taps possible.

The spec says two taps: Kind, then Context. The build asks Type → branch → maybe-detail, with chip groups that read like a taxonomy tree, not a quick gesture.

Symptom

For a graphic/artwork capture, the user faces a card list, a chip grid, then a four-field form with the keyboard up — before the photo is even saved.

Why it matters

The architect is walking. Each extra screen is another chance to give up and just leave the photo in the camera roll, untagged.

Issue 02 of 05

The second tap is hidden behind the first.

The spec's two-axis grid (Kind × Context) is replaced by a tree where the second axis depends on what you picked first.

Consequence

You can't see the whole vocabulary at once. Muscle memory never forms because the second screen keeps changing shape.

Issue 03 of 05

Chips read like form fields, not gestures.

Small grey pills with hairline borders are a desktop pattern. On a phone held one-handed they are precise targets, not bold gestures.

Hit-target audit

Most chips render at ~32–36 px tall. Apple's HIG and your own goal ("usable while walking") want 44 px minimum, ideally larger for the top-level choices.

Visual weight

Active and inactive chips look almost the same — a one-pixel border swap. The user has no sense of "I am committing to this."

Issue 04 of 05

The artwork detour blocks the save.

Picking Graphic → Artwork opens a four-field form (Title, Artist, Year, Source) and a "+ Add label photo" affordance — all before the photo is committed to the archive.

The trap

The user is now typing on a phone, in a museum, in front of the artwork. The very thing the spec calls "the enemy."

Recommendation

Save first. Move artwork details to the detail view, where they are already supported. The capture flow should never branch into a keyboard.

Issue 05 of 05

Gallery filters speak a different vocabulary.

Capture asks for Building / Element / Graphic. The browse filter offers exactly the same three. But the spec's browsing intent is along Context (Travel, Exhibition, Making, Reference) — which never gets captured.

Symptom

Two months from now the user wants "everything I shot in Venice on the trip." The data isn't there to filter on.

Recommendation

Either restore the Context tap from CLAUDE.md, or rely on auto-capture (date, GPS) and surface those in the filter row. Currently neither is happening.

Today's filter row

AllBuildingElementGraphic

Same axis as capture. Doesn't help when you remember where, not what.

CLAUDE.md, restated

"Tap one button, take photo, tag in 2 taps, done. The whole flow should take under 10 seconds."

The build today: 3–11 taps, 2–3 screens, sometimes a keyboard.

03 — DIRECTION

A return to two taps —
and what the menu could become.

Pattern proposal

One sheet. Both axes visible. No branching.

Bring up a single bottom sheet after the photo lands. Show the 3×4 grid the spec already defines. The user taps one cell — Kind and Context picked at once — and the sheet dismisses.

01

One screen, one gesture.

No "next." The grid commits on tap.

02

Both axes always visible.

Muscle memory forms after a week.

03

Skippable.

Tap "Save without tags" — photo lands in archive untagged, user enriches later in detail view.
